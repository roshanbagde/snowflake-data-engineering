Single Sign-On between Microsoft Entra ID (Azure AD) & Snowflake

A beginner-friendly, step-by-step guide to setting up SAML 2.0 SSO so users log into Snowflake with their Microsoft/Azure credentials.

IdP: Microsoft Entra ID SP: Snowflake Protocol: SAML 2.0 Works on free / trial accounts

What is this and why are we creating an Azure app? (read this first)

If this is your first time, this section gives you the full picture in plain language. Once it clicks, every step below will make sense.

The everyday problem we're solving

Normally, to log into Snowflake you create a separate username and password just for Snowflake. If your company uses 20 tools, that's 20 different passwords to remember, reset, and keep secure. When someone leaves the company, an admin has to remember to disable their account in all 20 places.

Single Sign-On (SSO) fixes this. Instead of Snowflake checking your password itself, it says: "I trust Microsoft. If Microsoft says you are who you claim to be, I'll let you in." So you log in once with your Microsoft/Azure account, and you get into Snowflake (and other apps) without typing a separate password.

The two players (a simple analogy)

🛂 Think of it like getting into an event using your passport

Identity Provider (IdP) = Microsoft Entra ID (Azure) → the passport office. It already knows you and can vouch for your identity.

Service Provider (SP) = Snowflake → the event venue. It doesn't issue IDs; it just checks the passport at the door.

The two players

Snowflake (the venue) doesn't want to manage everyone's identity. It would rather trust a party that already does that well — Microsoft (the passport office).

So what is the "Azure app" we create?

When you create the Snowflake app in Azure (called an Enterprise Application), you are not installing Snowflake software into Azure. You are creating a connection / agreement between Microsoft and Snowflake. That app is the place where you tell Microsoft:

"Hey Microsoft, there's an external service called Snowflake. Here's its address. When my users want to go there, here's how you should prove their identity to it, and here's which of my users are even allowed to use it."

In other words, the Azure app is the "trust bridge" that holds the configuration for the handshake between Microsoft and Snowflake. Without it, Microsoft has no idea Snowflake exists and wouldn't know how to vouch for your users.

What actually happens when a user logs in (the handshake)

1. User clicks "Azure SSO" on the Snowflake login page. 2. Snowflake says: "I don't know your password — go ask Microsoft to prove who you are." ↓ (redirect) 3. Microsoft (Entra ID) checks the user (password, MFA, etc.). 4. Microsoft creates a signed digital note: "Yes, this is jane.doe@company.com. I vouch for her." This signed note is called a SAML assertion. ↓ (redirect back, carrying the signed note) 5. Snowflake reads the note, checks Microsoft's signature is genuine, finds the matching Snowflake user, and logs Jane in. No Snowflake password needed.
The login handshake, step by step
👤 User1Clicks "Azure SSO" on the Snowflake login page
❄️ Snowflake2"I don't know your password — go prove who you are to Microsoft"  → redirects
🛂 Entra ID3Checks the user (password + MFA)
🛂 Entra ID4Creates a signed note (SAML assertion): "Yes, this is jane.doe@…"  → redirects back
❄️ Snowflake5Verifies the signature with the certificate, finds the matching user, logs Jane in ✅
No Snowflake password is ever used. The certificate proves step 4 really came from your Microsoft tenant.

The signature in step 4 is why we exchange a certificate later (so Snowflake can verify the note really came from your Microsoft tenant and wasn't forged). The email/name in the note is the NameID — that's how Snowflake knows which Snowflake account to open, which is why the user's identity must match on both sides.

Why two configurations (Azure side AND Snowflake side)?

Trust has to go both ways, like exchanging phone numbers:

  • In Azure, you tell Microsoft who Snowflake is (its URLs) and who is allowed in (Steps 2–5).
  • In Snowflake, you tell Snowflake to trust this specific Microsoft tenant by giving it Microsoft's address and certificate (Step 4).

That's the whole idea. Everything below is just filling in those details correctly.

Understanding certificates & secrets (and why you get those yearly renewal emails)

This explains, in plain language, what that certificate is, why your company gets an email to renew it every ~1 year, and what happens to logins if you ignore it.

What is the certificate, really?

Remember the handshake above: when a user logs in, Microsoft creates a signed note (the SAML assertion) that says "yes, this is Jane." The certificate is what makes that signature trustworthy.

🕯️ Think of it like a wax seal on a letter

Microsoft "stamps" every identity note with its private key (a secret only Microsoft has).

Snowflake holds the matching public certificate (the stamp's pattern) and checks every incoming note: "Does this seal match the pattern I was given? Yes → genuine. No → reject."

How the signature is trusted (matching key pair)

Only Microsoft's private key can make a seal its public certificate accepts. Expired/replaced cert  →  check fails  →  login rejected. This is the cert you paste into SAML2_X509_CERT.

So the certificate is how Snowflake knows the note really came from your Microsoft tenant and wasn't forged by an attacker. This is the same certificate you copy in Step 3c and paste into Snowflake as SAML2_X509_CERT in Step 4.

Where the user is actually shown as "verified"

The user's identity does not live in the certificate — it lives in the SAML assertion, the signed message Microsoft sends to Snowflake at login (step 4 of the handshake). Here is a trimmed, annotated example of what that message contains:

SAML assertion (the signed login message)<saml:Assertion>

  <!-- WHO vouched for the user: your Microsoft tenant (the IdP) -->
  <saml:Issuer>https://sts.windows.net/8a0e1f7c-1234-.../</saml:Issuer>

  <!-- WHO the user is — this is the NameID Snowflake matches to LOGIN_NAME -->
  <saml:Subject>
    <saml:NameID Format="...emailAddress">jane.doe@yourtenant.onmicrosoft.com</saml:NameID>
  </saml:Subject>

  <!-- WHEN it is valid (a few minutes) and WHO it is for (your Snowflake account) -->
  <saml:Conditions NotBefore="2026-06-06T10:00:00Z" NotOnOrAfter="2026-06-06T10:05:00Z">
    <saml:AudienceRestriction>
      <saml:Audience>https://ab12345.snowflakecomputing.com</saml:Audience>
    </saml:AudienceRestriction>
  </saml:Conditions>

  <!-- PROOF the user was actually authenticated (this is the "verified" part) -->
  <saml:AuthnStatement AuthnInstant="2026-06-06T10:00:00Z">
    <saml:AuthnContext>
      <saml:AuthnContextClassRef>
        urn:oasis:names:tc:SAML:2.0:ac:classes:PasswordProtectedTransport
      </saml:AuthnContextClassRef>
    </saml:AuthnContext>
  </saml:AuthnStatement>

  <!-- THE SEAL: this signature is created with Microsoft's private key and is
       checked by Snowflake using the public CERTIFICATE shown above. -->
  <ds:Signature>
    <ds:SignedInfo>...</ds:SignedInfo>
    <ds:SignatureValue>Qm9ndXNTaWduYXR1cmVWYWx1ZUZvckV4YW1wbGVPbmx5PT0=</ds:SignatureValue>
    <ds:KeyInfo>
      <ds:X509Data><ds:X509Certificate>MIIDpDCC...==</ds:X509Certificate></ds:X509Data>
    </ds:KeyInfo>
  </ds:Signature>

</saml:Assertion>

How to read it, in plain terms:

Part of the messageWhat it tells Snowflake
<Issuer>"This came from your Microsoft tenant."
<NameID>"The user is jane.doe@…." (must match Snowflake LOGIN_NAME)
<Conditions>"Only valid for the next ~5 minutes, and only for this Snowflake account."
<AuthnStatement>"Microsoft really did authenticate this person just now." ← the "verified" proof
<Signature>"And here's the seal — verify it with the certificate to be sure none of this was tampered with."

So the flow is: the AuthnStatement says the user is verified, the NameID says who they are, and the Signature (checked against the certificate) proves the whole note is genuine and unaltered. If the certificate has expired, that last check fails — and Snowflake rejects the login even though Microsoft really did verify the user.

People use the words interchangeably, but they're slightly different "keys":

For SSO, the thing that matters is the SAML signing certificate. If you also set up auto-provisioning (Appendix B) or other app integrations, those use secrets/tokens that expire too — same renewal story applies.

Why do they expire at all? Why not make them last forever?

Expiry is a security feature, not an inconvenience. Here's the simple reasoning:

  1. Limit the damage if a key leaks. If a signing key is ever stolen, an attacker could forge identity notes and impersonate your users. An expiry date means a stolen key is only useful for a limited time, not forever.
  2. Force regular "key changes." Just like you change important passwords periodically, rotating certificates regularly is good hygiene.
  3. Industry/compliance rules. Standards (and many company security policies) require keys to have a maximum lifetime — commonly 1 year — which is exactly why you get that annual email.

So the yearly renewal email is your company's system reminding you: "this key is about to reach its expiry date — replace it before it stops working."

What happens to authentication if the certificate EXPIRES?

⚠️ This is the important part

If the certificate expires and you didn't renew it, SSO logins break for everyone.

Here's why, using the handshake:

  • Microsoft keeps signing identity notes with its key.
  • But the certificate (stamp pattern) Snowflake holds is now expired/invalid.
  • Snowflake checks the seal, sees it's no longer valid, and rejects the login.
  • Users see errors like "invalid SAML response" or "signature validation failed" and cannot log in via SSO.

In short: expired certificate → the trust breaks → nobody can sign in through Azure SSO. This is why the renewal is treated as important — missing it can cause an outage where your whole team is locked out of Snowflake until it's fixed.

🛟 Safety net

This is exactly why Step 8 tells you to keep a password-based ACCOUNTADMIN backup login. If a certificate expires unexpectedly, that backup account lets an admin get in and fix the certificate without being locked out.

What exactly changes inside the certificate when it's renewed?

When Microsoft (or any IdP) renews the certificate, it does not just "extend the date" on the old one — it generates a brand-new certificate from scratch. A certificate is really a public key plus some details, bundled together and self-signed by Microsoft. Renewal makes a fresh bundle. Here's what's different and what stays the same:

Inside the certificateOld certNew certChanged?
Key pair (the public key + its secret private-key partner)key Akey B (freshly generated)✅ Yes — the big one
Valid from / Valid to datesthis yearnext year✅ Yes
Serial numberone valuea new value✅ Yes
Thumbprint (fingerprint / hash of the whole cert)hash Ahash B✅ Yes (everything above changed)
Signature value……✅ Yes
Subject / Issuer name (e.g. CN = Microsoft Azure Federated SSO Certificate)that textthe same text❌ Usually identical

The key insight: the most important change is the key pair. Microsoft generates a new private key (kept secret on Microsoft's side) and its matching new public certificate (the one you hand to Snowflake). From the moment the new cert goes active, Microsoft signs every login note (the SAML assertion) with the new private key.

Why that breaks logins if you don't update Snowflake

Snowflake is still holding the old public certificate, but new assertions are now signed with the new private key. The old public cert mathematically cannot verify a signature made by the new key → you get "signature validation failed." That is the entire reason the renewal steps below make you paste the new cert into SAML2_X509_CERT.

🔎 Why two certs can look identical

Because the Subject/Issuer name text is usually the same on both, the old and new certs look alike at a glance. The reliable way to tell them apart is the Thumbprint and the Valid to date — those are unique to each certificate. Azure shows the thumbprint so you can confirm exactly which one is active.

✅ What Microsoft does NOT change

It's still the same tenant, same Issuer URL (SAML2_ISSUER), same Login/SSO URL (SAML2_SSO_URL), and the same users. Only the signing key/certificate is swapped. So in Snowflake you normally only need to update SAML2_X509_CERT — not the issuer or SSO URL.

How to renew/rotate the certificate (the safe way, no downtime)

The trick is to add the new certificate before the old one expires, so there's an overlap where Snowflake accepts both. Then remove the old one.

Zero-downtime certificate rotation (overlap)
① Add NEW cert in Azure(leave inactive)
➜
② Put NEW cert in SnowflakeSAML2_X509_CERT
➜
③ Activate NEW in Azureboth valid briefly
➜
④ Test login
➜
⑤ Remove OLD cert
Overlapping the old & new certs means logins never break. Do it a week or two before the expiry date in the email.

On the Azure side

  1. Open the Snowflake enterprise app → Single sign-on → SAML Certificates section.
  2. Click Edit → New Certificate → set an expiry → Save.
  3. You now have two certs: the Active (old) and the Inactive (new). Don't activate yet.
  4. Download/copy the new certificate's Base64 value.

On the Snowflake side — update the integration with the new cert

sqlUSE ROLE ACCOUNTADMIN;

ALTER SECURITY INTEGRATION azure_entra_sso
  SET SAML2_X509_CERT = '<paste the NEW Base64 cert, no BEGIN/END lines>';

Then flip the new cert live in Azure

  1. Back in Azure, make the new certificate Active.
  2. Test a login (use an incognito window). If it works, you're done.
  3. Later (after confirming all is well), delete the old certificate in Azure.
✅ Best practice

Do the renewal a week or two before the expiry date in the email, not on the last day. Test immediately after. Keep the password backup admin ready just in case.

Quick "what to do when the renewal email arrives" checklist

  • Note the expiry date in the email — schedule the work a couple of weeks earlier.
  • Confirm you have a password-based ACCOUNTADMIN backup that still works (Step 9).
  • In Azure: create a New Certificate in the Snowflake app's SAML settings.
  • Copy the new Base64 cert → run ALTER SECURITY INTEGRATION ... SET SAML2_X509_CERT in Snowflake.
  • Make the new cert Active in Azure.
  • Test login in an incognito window (both IdP- and SP-initiated).
  • Remove/retire the old certificate once the new one is confirmed working.
  • (If you use SCIM/Appendix B) Also check whether the SCIM token/secret needs renewing.

What is "Entra ID P1 / P2"? (the term used throughout this guide)

Entra ID (formerly Azure AD) is sold in tiers. The basic identity service — letting people sign in and use SSO — is Free. The advanced security and automation features cost extra and are sold as Entra ID P1 and P2 (these are the same licenses bundled inside Microsoft 365 E3 and E5). So when this guide says "requires P1/P2," it just means "that feature is switched off unless your tenant has bought that plan."

🏨 Think of hotel room tiers

Free = a standard room (a bed — you can sleep = users can sign in & do SSO). P1 = an upgraded room with extras (Conditional Access rules, assigning whole groups to an app, and SCIM auto-provisioning). P2 = the full suite — everything in P1 plus a concierge watching for trouble (risk-based sign-in detection, PIM just-in-time admin, access reviews). Each tier includes everything below it.

4. Configure Snowflake to accept Entra ID (SAML IdP)

There are two ways. Option A (SECURITY INTEGRATION) is the modern, recommended method.

Option A — Create a SAML2 Security Integration (recommended)

Open a Snowflake worksheet as ACCOUNTADMIN and run:

sqlUSE ROLE ACCOUNTADMIN;

CREATE OR REPLACE SECURITY INTEGRATION azure_entra_sso
  TYPE = SAML2
  ENABLED = TRUE
  SAML2_ISSUER = '<Azure AD Identifier — from the Azure SAML page, box "4. Set up Snowflake" (collected in Step 3c)>'
  SAML2_SSO_URL = '<Login URL — from the Azure SAML page, box "4. Set up Snowflake" (collected in Step 3c)>'
  SAML2_PROVIDER = 'CUSTOM'
  SAML2_X509_CERT = '<paste Base64 cert body — no BEGIN/END lines, no line breaks>'
  SAML2_SP_INITIATED_LOGIN_PAGE_LABEL = 'Azure SSO'
  SAML2_ENABLE_SP_INITIATED = TRUE;

"Section 4" here means a box on the Azure portal, not Step 4 of this guide. On the Azure "Set up Single Sign-On with SAML" page, the boxes are numbered 1–4; the last box is titled "4. Set up Snowflake" and contains both values:

  • Azure AD Identifier → SAML2_ISSUER (looks like https://sts.windows.net/<tenant-id>/)
  • Login URL → SAML2_SSO_URL (looks like https://login.microsoftonline.com/<tenant-id>/saml2)

You already copied these in Step 3c above. Path: Entra ID → Enterprise applications → your Snowflake app → Single sign-on → box "4. Set up Snowflake".

Notes:

  • SAML2_X509_CERT = the contents of the Certificate (Base64) file from Azure box 3 (SAML Certificates), with the -----BEGIN CERTIFICATE----- / -----END CERTIFICATE----- lines removed (Snowflake tolerates the internal line breaks, but strip the header/footer).
  • SAML2_ENABLE_SP_INITIATED = TRUE adds an "Azure SSO" button on the Snowflake login page (service-provider-initiated flow).

Verify it:

sqlDESC SECURITY INTEGRATION azure_entra_sso;

sqlUSE ROLE ACCOUNTADMIN;

  "certificate": "<Base64 cert, no headers>",

Prefer Option A for new setups. Only use Option B if you have an older account that requires it.

5. Create a test user in Microsoft Entra ID

To validate SSO end-to-end you need a real identity on both sides. Start by creating a dedicated test user in Entra ID. (Skip this if you already have a user you want to test with — just note their email / UPN.)

5a. Where to create it

Users live in the directory itself, not inside the enterprise app:

  1. Azure portal → search for and open Microsoft Entra ID.
  2. Left menu → Users → All users.
  3. Click + New user → Create new user (not "Invite external user" — that's for guests).

5b. What details to fill in

6. Assign the user to the Snowflake app (Users and groups)

Even with SSO configured, a user cannot sign in to Snowflake through Azure until they're explicitly assigned to the enterprise app.

  1. Azure portal → Microsoft Entra ID → Enterprise applications → open your Snowflake SSO app.
  2. In the app's left menu, click Users and groups.
  3. Click + Add user/group at the top.
  4. Under Users, click None Selected → search for and tick your test user (e.g. Jane Doe) → Select.
  5. If the app defines app roles, a Select a role option appears — choose the default (often User) if prompted.
  6. Click Assign. The user now shows in the Users and groups list.

Users vs. groups: On the free Entra ID tier you can only assign individual users. Assigning a whole group to an app requires Entra ID P1/P2 (covered by the P2 trial). For this validation, assigning the single test user is enough.

Want everyone in the tenant to get the app without assigning each person? In the app's Properties, set Assignment required? to No. For a controlled test, leave it Yes and assign explicitly.

7. Make sure the user exists in Snowflake with a matching login

SSO authenticates the user, but the user must already exist in Snowflake, and their LOGIN_NAME must equal the SAML NameID you chose in Step 3b (the email or UPN). Use the exact same address you noted for the Entra test user in Step 5 — copy it verbatim.

sqlUSE ROLE ACCOUNTADMIN;

CREATE USER jdoe
  LOGIN_NAME = 'jane.doe@yourtenant.onmicrosoft.com'  -- must match the SAML NameID
  DISPLAY_NAME = 'Jane Doe'
  EMAIL = 'jane.doe@yourtenant.onmicrosoft.com'
  DEFAULT_ROLE = PUBLIC
  MUST_CHANGE_PASSWORD = FALSE;

-- optional: grant a usable role
GRANT ROLE PUBLIC TO USER jdoe;

If the NameID coming from Azure is the email but the Snowflake LOGIN_NAME is the username, the login will fail with "The user you were trying to authenticate as doesn't exist." Keep them identical.

8. Test the connection

Test 1 — IdP-initiated (from Azure)

  1. Back in the SAML SSO page (section 5) click Test (or use the Microsoft My Apps portal at https://myapps.microsoft.com and click the Snowflake tile).
  2. You should be redirected to Snowflake and logged straight in.

Test 2 — SP-initiated (from Snowflake)

  1. Go to your Snowflake login page https://<your-account>.snowflakecomputing.com.
  2. Click the "Azure SSO" button.
  3. Authenticate with Entra ID → you should land in Snowsight.

Use an incognito/private window for testing so you don't reuse an existing session.

9. (Recommended) Keep an admin password login as a backup

If SSO is misconfigured you can lock yourself out. Always keep at least one ACCOUNTADMIN user that can log in with a username/password (or key-pair), and don't disable password login account-wide until SSO is fully verified.

sql-- Confirm at least one ACCOUNTADMIN can still use password auth
SHOW USERS;

Appendix B — (Optional) Automatic user provisioning (SCIM)

What SCIM is and why you'd use it

SSO (the main guide) handles logging in — but it assumes the user already exists in Snowflake. SCIM handles the other half: creating, updating, and removing those users automatically, so you don't have to run CREATE USER by hand for every person.

🔄 Simple way to think about it

SSO = the door lock that checks who's allowed in. SCIM = the HR system that keeps the guest list up to date — adding new joiners, updating job changes, and removing leavers — automatically.

SSO vs SCIM — two different jobs
🚪 SSO (SAML)checks who's allowed in,
at login time
🔄 SCIMkeeps the guest list current:
add / update / remove users
🛂 Entra IDyou assign a user/group
push (SCIM token) ➜
❄️ Snowflakeuser auto-created;
leaver auto-disabled
SSO = the door lock. SCIM = the HR system that maintains who exists. Set up & verify SSO first, then add SCIM on top.

With SCIM turned on: when you assign a user (or group) to the Snowflake app in Azure, Azure creates the matching Snowflake user for you. When you unassign someone or they leave the company, Azure disables them in Snowflake. No manual SQL per user.

Requires Microsoft Entra ID P1 or P2 (the 30-day P2 trial works). Automatic provisioning is not available on the free Entra tier — SSO still is.

How SSO and SCIM work together

SSO (SAML)SCIM
JobAuthenticates a loginCreates / updates / removes the user record
DirectionUser → logs inAzure → pushes users into Snowflake
Secret usedSAML signing certificateSCIM bearer token
Needs paid license?No (free tier OK)Yes (Entra P1/P2)

You can run SSO without SCIM (just create users manually — Step 6). SCIM is purely a convenience layer on top. Set up and verify SSO first, then add SCIM.

Step 1 — Set up the Snowflake side

In a Snowflake worksheet as ACCOUNTADMIN, create a role for the provisioner, then the SCIM integration, then generate the token:

sqlUSE ROLE ACCOUNTADMIN;

-- 1. Role that Azure will "act as" when creating users/roles
CREATE OR REPLACE ROLE aad_provisioner;
GRANT CREATE USER ON ACCOUNT TO ROLE aad_provisioner;
GRANT CREATE ROLE ON ACCOUNT TO ROLE aad_provisioner;
GRANT ROLE aad_provisioner TO ROLE ACCOUNTADMIN;

-- 2. The SCIM integration itself
CREATE OR REPLACE SECURITY INTEGRATION aad_scim_provisioning
  TYPE = SCIM
  SCIM_CLIENT = 'AZURE'
  RUN_AS_ROLE = 'AAD_PROVISIONER';

-- 3. Generate the bearer token Azure will use (COPY IT NOW — shown only once)
SELECT SYSTEM$GENERATE_SCIM_ACCESS_TOKEN('AAD_SCIM_PROVISIONING');

The token from step 3 is displayed only once and is valid for 6 months. Copy it immediately and store it safely — you'll paste it into Azure next, and you'll need to rotate it before it expires (see token rotation below).

Step 2 — Set up the Azure side

  1. In the Snowflake enterprise app → Provisioning → Get started → set Provisioning Mode to Automatic.
  2. Under Admin Credentials, enter:
    • Tenant URL: https://<account>.snowflakecomputing.com/scim/v2/
    • Secret Token: the token from SYSTEM$GENERATE_SCIM_ACCESS_TOKEN above.
  3. Click Test Connection → it should report success → Save.
  4. Don't turn it On yet — first configure scope and mappings (below), then switch Provisioning Status to On.

Step 3 — Choose the scope (who gets synced)

Under Settings → Scope, pick one:

Scope optionMeaning
Sync only assigned users and groups (recommended)Only the people/groups you explicitly add under Users and groups are created in Snowflake. Safe and predictable.
Sync all users and groupsEveryone in your directory is pushed to Snowflake. Rarely what you want — avoid on shared tenants.

Stick with "assigned users and groups". It means the same Users and groups list you set up for SSO in Step 5 also controls who gets provisioned — one list, two purposes.

Step 4 — Attribute mappings (which fields map to what)

Under Mappings → Provision Azure Active Directory Users, Azure decides which Entra field becomes which Snowflake field. The defaults from the gallery app are usually correct. The ones that matter most:

Azure (Entra) attributeSnowflake attributeNotes
userPrincipalNameuserName → becomes Snowflake LOGIN_NAMEMost important. This must match the SAML NameID from SSO (Step 3b), or SSO login won't find the SCIM-created user.
mailemails[type eq "work"].value → EMAILUsed for notifications.
displayNamename.formatted / displayNameThe friendly name shown in Snowflake.
givenName / surnamename.givenName / name.familyNameFirst / last name.
Switch([IsSoftDeleted]…)activeControls enable/disable — drives deprovisioning (below).
⚠️ Keep SSO and SCIM identities aligned

SSO matches users by NameID; SCIM creates users with userName → LOGIN_NAME. If SSO sends the email but SCIM provisions the UPN (or vice-versa) and they differ, users get created but can't log in. Make both sides use the same identifier (usually userPrincipalName / email).

Step 5 — Group provisioning → Snowflake roles

If you enable Provision Azure Active Directory Groups, each assigned Entra group is created as a Snowflake role, and group membership is synced as role grants:

  • Entra group Snowflake-Analysts → Snowflake role SNOWFLAKE-ANALYSTS.
  • Adding a user to that Entra group → Azure grants them the role in Snowflake.

SCIM creates the role and the grants, but it does not grant privileges to that role (e.g. access to a warehouse or database). You still grant those once in Snowflake: GRANT USAGE ON WAREHOUSE … TO ROLE "SNOWFLAKE-ANALYSTS";. After that, membership stays automatic.

Step 6 — Deprovisioning (what happens when someone leaves)

This is a big reason to use SCIM. When a user is unassigned from the app, disabled, or deleted in Entra:

  • Azure sends an update setting the Snowflake user's active attribute to false.
  • Snowflake disables the user (sets DISABLED = TRUE) — it does not drop/delete them, so their history is preserved.
  • The disabled user can no longer log in via SSO.

Because deprovisioning disables rather than deletes, re-assigning the user later re-enables the same account. To fully remove a user you still run DROP USER manually in Snowflake.

Step 7 — Rotating the SCIM token (the yearly-renewal cousin)

Just like the SAML certificate expires, the SCIM bearer token expires (default 6 months). If it lapses, provisioning silently stops — new joiners won't get accounts and leavers won't get disabled. Rotate it before expiry:

sqlUSE ROLE ACCOUNTADMIN;

-- Generate a fresh token for the existing integration
SELECT SYSTEM$GENERATE_SCIM_ACCESS_TOKEN('AAD_SCIM_PROVISIONING');
  1. Copy the new token.
  2. In Azure → Snowflake app → Provisioning → Admin Credentials → paste it into Secret Token.
  3. Test Connection → Save.

Set yourself a reminder ~5 months out. This is the same "renew the secret" habit as the SAML certificate — just a different secret. See the certificates & secrets section.
