☁️ 1. What is Azure & the Azure Portal?

Microsoft Azure is a cloud platform — a giant network of Microsoft data centers around the world. Instead of buying physical servers, you "rent" computing power, storage, and databases on demand and pay only for what you use.

Cloud, explained simply

🏢 Old way
Buy servers,
maintain them
➜
☁️ Cloud way
Rent on demand,
Microsoft maintains
You stop worrying about hardware and focus on building.

The Azure Portal (https://portal.azure.com) is the website / control panel where you click buttons to create and manage everything. There are other ways to control Azure (CLI, code), but the portal is the visual, beginner-friendly one.

TermPlain meaning
Azure PortalThe web dashboard you log into to manage Azure
ResourceAny single thing you create (a database, a storage account, a VM)
ServiceA category/product of resources (e.g. "Azure SQL", "Storage")
SubscriptionThe "billing folder" — where charges land
Tenant / Entra IDYour organization's identity directory (users & logins)
✅ Your trial The free trial usually gives you ~$200 credit for 30 days plus several "always free" services. Watch the credit meter in Cost Management (Section 12) so you don't get surprised.

LevelWhat it isReal-world analogy

📁 4. Resources & Resource Groups

A resource group (RG) is just a folder. But choosing a good folder strategy keeps your account tidy and makes cleanup (and cost tracking) easy.

Good resource group habits

  • Group by lifecycle/project — put everything for one project in one RG so you can delete the whole RG when done (great for trials!).
  • One region per RG is common but not required. The RG itself has a region (it stores metadata).
  • Naming convention — e.g. rg-dataeng-dev-eastus. Consistency helps a lot.
  • Tags — key/value labels like env=dev, owner=roshan, project=learning for filtering & cost reports.
📁 rg-learning-dev
⤵
🗄️ Storage
🛢️ SQL DB
🏭 Data Factory
🔐 Key Vault
One project folder holding all related resources. Delete the folder → all of it goes (and billing stops).
⚠️ Cost safety on a trial When you finish an experiment, delete the entire resource group. This is the cleanest way to stop charges and avoid forgotten resources eating your credit.

Creating a resource — the universal flow

  1. Click + Create a resource (or search the service name).
  2. Pick the Subscription and Resource group (create a new one if needed).
  3. Choose a Name, Region, and a tier/SKU (size = price).
  4. Step through tabs: Networking, Security, Tags…
  5. Review + Create → Azure validates → Create. A "deployment" runs.

🌍 5. Regions & Availability

A region is a geographic location of Azure data centers (e.g. East US, Central India, West Europe). You pick a region for almost every resource.

📍Region

A set of data centers in one area. Pick one close to your users for speed.

🏢Availability Zone

Physically separate buildings within a region for resilience.

🌐Geography

A group of regions tied to data-residency rules (e.g. EU, India).

🔁Region Pair

Two regions linked for backup/disaster recovery.

💡 How to choose ① closeness to users (lower latency), ② data-residency/legal rules, ③ which services are available there, ④ price (regions differ a little).
✅ Keep things together Put resources that talk to each other (e.g. Data Factory + Storage + SQL) in the same region to reduce latency and avoid cross-region data-transfer fees.

🆚 App Registration vs Enterprise Application — the deep dive

This is the #1 confusing pair in Entra ID, because they're closely related and the portal gives each its own menu. Once the mental model clicks, it's actually simple. Let's build it up slowly.

The one-sentence version

🔑 In plain English
App Registration = the definition / blueprint of an app — created in the tenant that owns the app.
Enterprise Application = the local instance of an app that is allowed to run inside your tenant (this instance is technically called a service principal).

🏪 The franchise restaurant analogy

Imagine a restaurant chain like "Pizza Hut":

📐 App Registration
The global brand
+ recipe book
(the blueprint)
➜ creates ➜
🏪 Enterprise Application
The actual branch
open in YOUR town
(the local instance)
One blueprint (registration) → one branch (enterprise app / service principal) in each town (tenant) where the brand operates.
  • App Registration = head office + master recipe. It defines what the restaurant is: name, logo, menu, and the rules a branch follows. It exists once, in the brand's home tenant.
  • Enterprise Application = a physical branch. When Pizza Hut opens in your town, that local branch is a separate, real thing you manage locally — set its opening hours, decide who's allowed in, fire a bad manager — without changing the global brand.
  • One blueprint can spawn many branches — one in every town (tenant). Microsoft Teams has one app registration (Microsoft owns it) but a separate enterprise application appears in every company's tenant that uses Teams.

🛂 Another analogy: passport vs visa

Passport vs Visa
The App Registration is the app's passport — issued by its home country (home tenant), it proves "this is who I am" globally.
The Enterprise Application is the visa / entry stamp in your country (your tenant) — "this specific app is permitted to operate here, with these permissions, for these people."

So when do I use which? (the practical rule)

If you want to…You use…Where in portal
Build your own app/script that signs in users or calls Azure APIsApp Registration (auto-creates a matching Enterprise App too)Entra ID → App registrations → + New registration
Connect a ready-made SaaS app for SSO (Snowflake, Salesforce, Tableau, ServiceNow…)Enterprise Application (add from the gallery)Entra ID → Enterprise applications → + New application
Give a script / CI-CD pipeline / daemon a "robot login" to AzureApp Registration (creates a service principal with a client secret/cert)Entra ID → App registrations → + New registration
Control who is allowed to use an app, or turn on SCIM provisioningEnterprise Application (Users & groups / Provisioning blades)Entra ID → Enterprise applications → pick the app
Manage the app's credentials, redirect URIs, API permissions, secretsApp RegistrationEntra ID → App registrations → pick the app
🔑 The golden rule
Asking "what IS this app and what can it request?" (secrets, redirect URLs, API permissions) → go to App Registration.
Asking "is this app allowed in MY org, and who can use it?" (assign users, SSO, provisioning, consent) → go to Enterprise Application.

How they relate (very important)

📐 Application object (App Registration)
Lives in the home tenant. The global definition. There is exactly ONE.
🏪 Service Principal (Enterprise App) — Tenant A
One local instance per tenant that uses the app.
🏪 Service Principal (Enterprise App) — Tenant B
A separate instance — managed independently.
🏪 Service Principal (Enterprise App) — Tenant C
…and so on, one per tenant.
1 App Registration (blueprint) → many Enterprise Applications (one local instance / service principal per tenant).
💡 Why you sometimes see BOTH for the same app
When you create an App Registration in your own tenant, Azure automatically creates a matching Enterprise Application (service principal) in the same tenant. That's why your custom app shows up in both menus — they're two faces of the same app: the registration face (define it) and the enterprise face (run & govern it). For a third-party SaaS like Snowflake you usually see only the Enterprise Application, because Snowflake owns the registration in their tenant — you just get the local instance.

🛠️ Two concrete examples you'll actually hit

Example A — Connecting Snowflake for SSO (a SaaS app you don't own)

Snowflake already built and registered their app in Snowflake's own Entra tenant. You're not building anything — you just want your employees to log into Snowflake with their company account.

  1. Go to Entra ID → Enterprise applications → + New application.
  2. Search the gallery for "Snowflake" and add it. → An Enterprise Application now exists in your tenant.
  3. Open it → Single sign-on → configure SAML. Then Users and groups → assign who can use it. Optionally Provisioning → set up SCIM.

You never touched App registrations. Everything happened on the Enterprise Application side, because you're consuming someone else's app, not defining one.

Example B — A Python script that reads secrets from Key Vault (an app you DO own)

You're writing your own automation. It needs a non-human "robot identity" to authenticate to Azure.

  1. Go to Entra ID → App registrations → + New registration. Name it kv-reader-bot.
  2. Under Certificates & secrets, create a client secret (or upload a certificate). Note the Client ID + Tenant ID.
  3. Azure auto-created a matching Enterprise Application (the service principal). Give that identity an RBAC role (e.g. Key Vault Secrets User) on your Key Vault.
  4. Your script signs in with Client ID + secret + Tenant ID → gets a token → reads the secret.

Here you started on the App registration side (define the identity + credentials), and the Enterprise Application side is where that identity gets permissions. Both faces, one app.

📦 Entra ID licence tiers: Free, P1 & P2 (the terms you keep seeing)

Entra ID comes in tiers (paid plans). The basic identity service is Free, but the advanced security & governance features cost extra — sold as Entra ID P1 and P2 (these are the licences bundled inside Microsoft 365 E3 and E5 respectively). When a doc says "requires Entra ID P1," it means "this feature is off unless you've bought that plan."

🏨 Simple analogy: hotel room tiers Free = a standard room (a bed — you can sleep = users can sign in). P1 = an upgraded room with extras (gym + breakfast = Conditional Access, SSO at scale, self-service password reset). P2 = the full suite — everything in P1 plus the concierge that watches for trouble (risk detection, PIM just-in-time admin, access reviews). Each tier includes everything below it.

Who + What + Where = one role assignment. Made under "Access control (IAM)".

The roles you'll use 90% of the time

RoleCan doUse for
OwnerEverything, including granting access to othersYou, the account owner
ContributorCreate/manage resources, but can't grant accessEngineers building stuff
ReaderView only, no changesAuditors, stakeholders
User Access AdministratorManage others' access onlyAccess managers
Data-plane roles (act on data inside a resource, not the resource itself):
Storage Blob Data ContributorRead/write files in a storage accountData engineers reading/writing blobs
SQL DB ContributorManage SQL databasesDBAs
Key Vault Secrets UserRead secrets from Key VaultApps fetching passwords
🔑 Control plane vs Data plane
Control plane = managing the box (create/delete a storage account).
Data plane = touching what's inside (read the actual files). You often need separate roles for each! A classic gotcha: you're "Owner" but still can't read blobs until you also get "Storage Blob Data Reader."

How to assign a role (the IAM flow)

  1. Open the scope (a subscription, RG, or resource).
  2. Click Access control (IAM) in the left menu.
  3. + Add → Add role assignment.
  4. Pick the Role → choose members (user/group/identity) → Review + assign.
✅ Best practice Assign roles to groups, not individuals. Grant the least privilege needed, at the narrowest scope that works.

🔑 8. SSO — Single Sign-On

Single Sign-On (SSO) lets users log in once with their company account and then access many apps (Snowflake, Power BI, Salesforce, Tableau…) without typing more passwords. Entra ID becomes the central "identity provider."

👤 User logs in once
to Entra ID
➜
🎫 Gets a token
❄️ Snowflake
📊 Power BI
☁️ Salesforce
📈 Tableau
One login → token → access to all connected apps. No separate passwords per app.

Key SSO vocabulary

TermMeaning
IdP (Identity Provider)The system that verifies identity = Entra ID in your case
SP (Service Provider)The app you're logging into (Snowflake, etc.)
SAML 2.0An older but very common SSO protocol (XML-based). Used by Snowflake, many SaaS.
OIDC / OAuth 2.0Modern token-based protocols (JSON). Used by Power BI & most new apps.
Metadata / FederationThe config file/URLs the two sides exchange to trust each other
Claims / AttributesInfo in the token (email, name, groups) sent to the app

Setting up SSO for an app — general steps

  1. Go to Entra ID → Enterprise applications → + New application.
  2. Search the gallery (e.g. "Snowflake") or add a custom/non-gallery app.
  3. Open the app → Single sign-on → choose SAML (or OIDC).
  4. Fill the Identifier (Entity ID) and Reply URL (ACS URL) from the target app.
  5. Configure attributes & claims (which user info to send).
  6. Download Entra's Federation Metadata XML / certificate and paste into the app.
  7. Assign users/groups who are allowed to use it, then Test.
💡 SSO ≠ provisioning SSO controls logging in. It does not create the user account inside the app. Creating/updating/deleting accounts automatically is SCIM provisioning — the next section. They're usually set up together.

🔄 9. SCIM Provisioning

SCIM (System for Cross-domain Identity Management) is the standard that lets Entra ID automatically create, update, and deactivate user accounts inside other apps. It keeps everything in sync — no manual account creation.

🆔 Entra ID
source of truth
➜ SCIM ➜
🎯 Target app
Snowflake, etc.
A new hire added in Entra → account auto-appears in the app. Someone leaves → account auto-disabled.

What SCIM keeps in sync

➕Create

New user in Entra → account auto-created in the app.

✏️Update

Name/email/role changes flow through automatically.

➖Deprovision

User removed/disabled in Entra → access revoked in the app.

👥Group sync

Group membership can map to roles in the target app.

Setting up SCIM — general steps

  1. In the Enterprise application → open Provisioning.
  2. Set Provisioning Mode = Automatic.
  3. Enter the app's Tenant URL (SCIM endpoint) and a Secret Token (from the target app).
  4. Click Test Connection to confirm Entra can reach it.
  5. Configure attribute mappings (which Entra fields map to which app fields).
  6. Set Scope = "assigned users and groups only" → Save → turn Provisioning ON.
SSO vs SCIM — don't mix them upSSOSCIM
PurposeLet users log inManage accounts (create/update/remove)

✅ Best practice Configure SSO + SCIM together for any business app. SSO = the door; SCIM = issuing & revoking the keys automatically.

🚦 10. Conditional Access & MFA

Multi-Factor Authentication (MFA) requires a second proof of identity (phone app, code) beyond a password. Conditional Access is the rules engine that decides when to demand MFA or block access.

👤 Sign-in attempt
➜
🚦 Conditional Access
evaluates signals
➜
✅ Allow
🔐 Require MFA
📍 Block risky location
💻 Require managed device
"IF [user/location/device/risk] THEN [allow / require MFA / block]."

A Conditional Access policy is an IF → THEN rule:

  • Assignments (the IF): which users, which apps, conditions (location, device, sign-in risk).
  • Access controls (the THEN): Grant with MFA, block, require compliant device, etc.
Entra ID → Security → Conditional Access → + New policy
✅ Easy win Turn on Security Defaults (or a CA policy) to require MFA for admins. It's the single biggest security improvement you can make.
💡 Licensing Full Conditional Access needs Entra ID P1/P2. Basic Security Defaults are free and cover the essentials for a trial.

🔐 14. Security & Key Vault

Security is layered. Here are the must-know pieces.

🗝️Key Vault

Secure store for secrets (passwords/connection strings), keys (encryption), and certificates. Apps fetch secrets instead of hardcoding them.

🛡️Defender for Cloud

Security posture score + recommendations + threat alerts across resources.

🤖Managed Identity

Password-free identity for a resource so it can access Key Vault/Storage securely — no secrets to manage.

🌐Private Endpoint

Give a service a private IP inside your network so traffic never touches the public internet.

🔒Encryption

Data is encrypted at rest & in transit by default. You can bring your own keys.

🚪Firewall rules

Restrict which IPs/networks can reach a database or storage account.

🔑 The golden pattern for data engineering
Resource gets a Managed Identity → that identity is given an RBAC role on Storage/Key Vault → it reads secrets from Key Vault. Result: no passwords anywhere in your code. You'll use this constantly with Data Factory, Databricks, and Synapse.

🗄️ 17. Storage Accounts & ADLS Gen2

The storage account is the foundation of nearly every data project. ADLS Gen2 (Azure Data Lake Storage) is a storage account with the "hierarchical namespace" turned on, which makes it behave like a real folder-based file system — ideal for analytics.

The 4 storage services inside one account

📦Blob / Containers

Object storage for any file (CSV, Parquet, images). The data lake lives here.

📁Files

Managed network file shares (SMB/NFS).

📨Queues

Simple message queue for app-to-app messaging.

🗂️Tables

Cheap NoSQL key-value store.

Things you'll configure

SettingMeaning
Performance: Standard / PremiumStandard = cheap HDD-backed; Premium = fast SSD for low latency
Redundancy (LRS/ZRS/GRS)How many copies & where. LRS = 3 copies in one DC; GRS = copies in another region
Access tier (Hot/Cool/Archive)Hot = frequent access; Cool/Archive = cheaper for rarely-used data
Hierarchical namespaceTurn ON = ADLS Gen2 (folders, fast analytics). Set at creation only!

How to access the data

  • Portal → Storage browser, or Azure Storage Explorer (desktop app).
  • RBAC roles — give yourself Storage Blob Data Contributor to read/write files (remember: control vs data plane!).
  • SAS token — a time-limited URL with specific permissions for sharing.
  • Access keys — full-access master keys (avoid sharing; rotate regularly).
✅ For analytics always create an ADLS Gen2 (hierarchical namespace ON) and store data in Parquet/Delta format for speed and lower cost.

🛢️ 18. Azure SQL & Databases

Azure offers several relational database options. Know the difference so you pick the right one.

OptionBest forNotes
Azure SQL DatabaseMost new cloud appsFully managed single DB; serverless option auto-pauses to save money
SQL Managed InstanceLift-and-shift from on-prem SQL ServerNear-100% SQL Server compatibility
SQL Server on a VMFull OS control / legacy needsYou manage patching & backups (IaaS)
PostgreSQL / MySQL / MariaDBOpen-source app stacksManaged "Flexible Server" options

Key concepts when creating Azure SQL

  • Logical server — a container/endpoint (yourname.database.windows.net) that holds databases & sets admin login + firewall.
  • Purchasing model — DTU (simple bundled) or vCore (flexible, pick CPU/RAM). vCore + serverless is great for trials (auto-pause = pay near-zero when idle).
  • Firewall — add your client IP, and tick "Allow Azure services" so other Azure resources can connect.
  • Authentication — SQL login and/or Entra ID auth (preferred — central identity).
✅ Trial-friendly Pick vCore → Serverless with auto-pause. You'll barely spend credit while learning. Use the built-in Query editor in the portal to run SQL without installing tools.

❄️ 23. Snowflake Connectivity

Snowflake is a popular cloud data warehouse. You'll often need Azure and Snowflake to talk — for data movement, querying, identity, and security. Here are the main connection patterns.

① Data movement (ADF ⇄ Snowflake)

🗄️ ADLS / Blob
⇄
🏭 Data Factory
Snowflake connector
⇄
❄️ Snowflake
ADF has a native Snowflake connector to copy data in/out, staging through Azure Storage.
  • In ADF, create a Linked Service → Snowflake (account URL, warehouse, DB, user/key).
  • ADF stages data through an Azure Storage account for fast bulk load (Snowflake COPY).
  • Snowflake's external stage can point directly at your ADLS Gen2 container.

② Identity / SSO (Entra ID → Snowflake)

  • Add Snowflake from the Entra Enterprise applications gallery.
  • Configure SAML SSO so users log into Snowflake with their Entra account (Section 8).
  • Add SCIM provisioning so Entra auto-creates/updates Snowflake users & roles (Section 9). Snowflake provides the SCIM endpoint & token.

③ Secure networking (Private Link)

  • Use Azure Private Link so traffic between Azure and Snowflake stays off the public internet.
  • Store Snowflake credentials/keys in Key Vault, accessed via Managed Identity.

④ Querying / analytics

  • Power BI has a native Snowflake connector (DirectQuery or Import) — see next section.
  • Databricks/Synapse can read Snowflake via the Spark Snowflake connector.
💡 Snowflake runs on Azure too You can host a Snowflake account in an Azure region. Keeping it in the same region as your Azure storage reduces latency and egress costs.

🧪 26. Your Trial Lab Plan (hands-on path)

A suggested order to explore your trial safely. Each step builds skills from this guide. Delete each resource group when done.

  1. Set a Budget with alerts in Cost Management (do this first!).

  5. Turn on Security Defaults (MFA) and look at Conditional Access.

  1. Go to Enterprise applications; add a gallery app (e.g. a free SaaS or Snowflake trial).
  2. Configure SAML SSO and assign your test user.
  3. If the app supports it, set up SCIM provisioning and watch a user auto-create.
  4. Review the provisioning logs & sign-in logs.
Week 3 — Data Engineering Core
  1. Create an ADLS Gen2 storage account; upload a sample CSV.
  2. Create an Azure SQL Database (vCore serverless) and use the Query editor.
  3. Create a Data Factory; build a Copy pipeline CSV → SQL table.
  4. Store the SQL password in Key Vault; reference it from ADF.
  5. (Optional) Spin up a tiny Databricks single-node cluster & read the CSV (then terminate!).
Week 4 — Migration & Connectivity
  1. Try AzCopy to bulk-upload files to Blob.
  2. Explore Azure Migrate & DMS (just the setup screens to learn the flow).
  3. If using Snowflake: create an ADF Snowflake linked service and copy a table.

  5. Publish to a Power BI workspace and set up scheduled refresh.
